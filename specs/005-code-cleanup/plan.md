## Implementation Order

Order matters to avoid breakage mid-way:

1. **Add `base.css` to all mini-app HTML files first** — before removing CSS rules, so there's never a moment where styles are missing
2. **Remove duplicated CSS rules** from the four mini-app CSS files
3. **Delete `phonics-audio.js`** — file is already unloaded, deletion is safe
4. **Remove 26 `<audio>` tags from `abc/index.html`** — already confirmed `abc.js` has zero references to them
5. **Fix `sound.js` `var` → `const`** — single-character semantic change, no logic impact
6. **Fix HTML indentation** — `monster/index.html` and `space/index.html`
7. **Clean router.js stub comments** — cosmetic, lowest risk last

---

## Risk Assessment

| Change | Risk | Mitigation |
|--------|------|-----------|
| Remove `<audio>` tags from `abc/index.html` | Low | Confirmed `abc.js` has zero `getElementById('snd-*')` references |
| Delete `phonics-audio.js` | Low | Confirmed not loaded by any HTML file |
| Add `base.css` before removing CSS | Low | Stylesheet load order: `base.css` loads first, mini-app CSS after — existing specificity preserved |
| Remove duplicate CSS rules | Medium | Must diff each rule against `base.css` to confirm no mini-app-specific overrides exist |
| `var` → `const` in `sound.js` | Low | `const` IIFE is identical at runtime; all callers use `Sound.*` namespace |
| Indentation fixes | Low | Cosmetic only; no logic changes |
| Remove router.js comments | Low | Comments only; logic unchanged |

**Medium risk item — CSS diff procedure**: Before removing any selector from a mini-app CSS file, compare its property values against the `base.css` definition. If they are identical → safe to delete. If they differ → preserve as an override with a `/* mini-app override */` comment.
