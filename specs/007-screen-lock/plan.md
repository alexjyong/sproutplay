# Implementation Plan: Screen Lock — Global App Pinning

**Branch**: `007-screen-lock` | **Date**: 2026-06-06 | **Spec**: [spec.md](spec.md)

## Summary

Implement Android system-level screen pinning for the entire SproutPlay app. When a parent enables the lock, the app is pinned at the OS level — the child cannot exit via home button, recent apps, or back button. The parent unlocks by tapping a visible lock icon 4 times quickly (matching the babbypaint pattern). This is a **global** lock covering all of SproutPlay, not individual mini-apps.

---

## Technical Context

**Language/Version**: JavaScript ES6+ (vanilla), HTML5, CSS3, Java (Android)
**Primary Dependencies**: Capacitor 8 (existing), custom `ScreenPinningPlugin` (hand-written Java, copied from babbypaint)
**Storage**: localStorage via existing `Settings` IIFE module (`settings.js`)
**Testing**: Manual — enable lock, pin the app, test all exit methods, verify 4-tap unlock
**Target Platform**: Android (Capacitor-wrapped WebView)
**Project Type**: Mobile app (Capacitor hybrid)
**Performance Goals**: Screen pinning activates within 500ms of lock icon tap
**Constraints**: Vanilla JS only; no bundler; IIFE singleton pattern preserved; custom Java plugin (no npm package); must match babbypaint unlock pattern

**Existing infrastructure to reuse**:
- `babbypaint/app/android/app/src/main/java/dev/alexjyong/babbypaint/ScreenPinningPlugin.java` — Custom Capacitor plugin (2 methods: `enterPinnedMode`, `exitPinnedMode`). Direct copy.
- `babbypaint/app/www/js/index.js` (lines 195–280) — 4-tap unlock sequence logic, toast feedback, lock/unlock button state management. Reference implementation.
- `app/www/js/settings.js` — Settings persistence module (add `setScreenLocked()` and `isScreenLocked()`)
- `app/www/js/router.js` — `handleBackButton()` (block exit when locked)
- `app/www/index.html` — Hub header with settings gear (add lock icon next to it)

**Structure Decision**: The lock is implemented as a new `lock.js` IIFE module (following the existing pattern of `sound.js`, `settings.js`, `router.js`). It manages the lock icon UI, 4-tap sequence, toast feedback, and calls the custom `ScreenPinning` plugin. The plugin is a hand-written Java file copied from babbypaint — no npm package needed.

## Implementation Order

Order matters to avoid breakage mid-way:

1. **Create `ScreenPinningPlugin.java`** — copy from babbypaint, register in `MainActivity.java`
2. **Create `lock.js`** — the core lock logic (4-tap sequence, toast feedback, pin/unpin)
3. **Add lock icon HTML to `index.html`** — visible in hub header
4. **Add CSS to `base.css`** — lock icon styles (small circular button)
5. **Wire `lock.js` into `app.js`** — connect lock icon to pin/unpin actions
6. **Add lock state to `settings.js`** — `setScreenLocked()`, `isScreenLocked()`
7. **Block hub exit in `router.js`** — when locked, `handleBackButton()` returns `true`
8. **Verify all mini-apps** — test lock/unlock from hub and each mini-app

## Risk Assessment

| Change | Risk | Mitigation |
|--------|------|-----------|
| Create `ScreenPinningPlugin.java` (Java) | Medium | Direct copy from babbypaint — proven to work. Must register in `MainActivity.java`. |
| Register plugin in `MainActivity.java` | Low | Simple one-line change: `registerPlugin(ScreenPinningPlugin.class)`. |
| Create `lock.js` IIFE module | Low | Follows existing IIFE pattern. Self-contained DOM management. |
| Add lock icon to `index.html` | Low | Simple HTML element in existing header. No layout impact when hidden. |
| Add CSS to `base.css` | Low | New selectors only. No existing rules modified. |
| Wire lock icon in `app.js` | Low | Follows existing pattern for settings toggle wiring. |
| Add lock state to `settings.js` | Low | Follows existing pattern (set/get methods). |
| Block hub exit in `router.js` | Medium | Must handle: locked/unlocked states, lock icon visibility. Thorough manual testing per quickstart.md. |

**Medium risk item — `router.js` integration**: When locked, `handleBackButton()` on hub must return `true` (block exit) instead of `false`. This is the single point where Android would normally exit the app. Testing plan: verify back from hub with lock on (blocked), back from hub with lock off (normal), back from mini-apps with lock on (blocked), back from mini-apps with lock off (normal).
